# TypedProgram soundness matrix — bounded slice 1

> **Status: historical — C++/MLIR was retired in v0.5 (commit 842842bd). Current implementation: hew-codegen-rs direct Rust/Inkwell LLVM emission.**
> This matrix records the retired MessagePack-to-C++ backend boundary. Use it
> as certification history, not as the current v0.5 codegen contract.

This artifact maps the first bounded certification slice for checker-owned `TypedProgram` side tables that are already on the wire.

## Scope

Included in this slice:

- `expr_types`
- `method_call_receiver_kinds`
- `assign_target_kinds`
- `assign_target_shapes`
- `lowering_facts`

Explicitly deferred from this slice: every other `TypedProgram` field.

## Shared boundary points

For all five fields, the Rust frontend hands the selected side tables to codegen in the same two places:

- `hew-compile/src/lib.rs` — `compile_file` and `compile_program` build `FrontendArtifacts`, calling:
  - `enrich_program_ast_with_diagnostics(...)` for `expr_type_entries` and `method_call_receiver_kinds`
  - `build_assign_target_kind_entries(...)`
  - `build_assign_target_shape_entries(...)`
  - `build_lowering_fact_entries(...)`
- `hew-serialize/src/msgpack.rs` — `TypedProgram` is serialized with named wire keys and consumed by the C++ reader.
- `retired C++ backend MLIRGen.cpp` — the loaded vectors are copied into lookup maps before lowering:
  - `exprTypeMap`
  - `methodCallReceiverKindMap`
  - `assignTargetKindMap`
  - `assignTargetShapeMap`
  - `loweringFactMap`

## Cross-cutting findings

1. `expr_types`, `assign_target_kinds`, `assign_target_shapes`, and `lowering_facts` each have an identifiable checker-side output-boundary authority step.
2. `method_call_receiver_kinds` now has a checker-side post-pass validator (`validate_method_call_receiver_kinds_output_contract`) that removes entries referencing unknown type or trait names; graduated to full boundary-certified status in a follow-on pass.
3. All five fields are parsed by `retired C++ backend msgpack_reader.cpp`, but coverage is mostly indirect through `retired C++ backend integration tests` rather than field-specific reader unit tests.
4. The `handle_ownership_kinds` row is the planned soundness target for the
   handle-safety spec at `docs/specs/handle-safety-and-resource-lifetime.md`;
   none of its boundary points exist on the wire yet. Recording it here ahead
   of implementation makes the boundary explicit so that producer, validator,
   serializer, reader, and codegen consumer must all land together rather
   than as silent partial wiring.
5. The `remote_operation_typed_failure` row records the spec-first distributed
   requirement that remote operations preserve typed failure across future
   language, serialization, and codegen boundaries instead of collapsing to
   sentinel or zero-valued success.
